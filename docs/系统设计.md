# BTC Quant 量化交易系统设计文档

## 1. 系统概述

### 1.1 项目简介

BTC Quant 是一个基于深度学习的比特币量化交易系统，采用微服务架构设计。系统通过分析历史 K 线数据，提取技术指标特征，使用 TCN（时间卷积网络）模型预测交易信号，并基于预测结果自动生成交易决策。

### 1.2 核心特性

- **数据库优先查询策略**：优先从本地 SQLite 数据库获取数据，缺失数据自动从 Binance API 补充
- **多输出回归模型**：同时预测盈亏比、止损百分比、止盈百分比
- **时间衰减样本权重**：近期数据权重更高，适应市场变化
- **统一配置管理**：使用 YAML 配置文件，支持环境变量覆盖
- **微服务架构**：各模块独立部署，通过 HTTP API 通信

### 1.3 技术栈

| 组件 | 技术选型 |
|------|----------|
| 后端框架 | FastAPI |
| 深度学习 | PyTorch + TCN |
| 数据处理 | Pandas + NumPy |
| 数据库 | SQLite |
| 容器化 | Docker + Docker Compose |
| 配置管理 | YAML + 环境变量 |

### 2.3 数据流

1. **数据获取**：Data Service 从 Binance API 获取 K 线数据，存入 SQLite
2. **特征计算**：Features Service 接收 K 线数据，计算技术指标和结构特征
3. **模型预测**：Predict Service 使用特征进行推理，输出交易信号
4. **策略执行**：Strategy Service 根据信号执行交易（回测/模拟/实盘）
5. **系统评估**：Evaluation Module 计算系统和模型的各项指标

## 3. 核心设计

### 3.1 数据库优先查询策略

```python
def get_klines(symbol, interval, start_time, end_time):
    # 1. 首先查询本地数据库
    db_klines = db.query_klines(symbol, interval, start_time, end_time)
    
    # 2. 检查数据完整性，找出缺失的时间段
    missing_ranges = find_missing_ranges(db_klines, start_time, end_time, interval)
    
    # 3. 对于缺失数据，从 Binance API 获取
    for range_start, range_end in missing_ranges:
        api_klines = fetcher.fetch_klines(symbol, interval, range_start, range_end)
        db.insert_klines(api_klines)  # 保存到数据库
        all_klines.extend(api_klines)
    
    # 4. 返回完整数据集
    return sorted(all_klines, key=lambda x: x.timestamp)
```

### 3.2 标签生成

对于每个时刻 t，查看未来 window 个周期的数据：

```python
max_high = future_slice["high"].max()
min_low = future_slice["low"].min()

upside = max_high - current_close
downside = current_close - min_low

if upside > downside:
    direction = 1  # LONG
    tp_pct = upside / current_close
    sl_pct = downside / current_close
else:
    direction = -1  # SHORT
    tp_pct = downside / current_close
    sl_pct = upside / current_close

rr = tp_pct / sl_pct * direction
```

### 3.3 时间衰减样本权重

```python
Weight = exp(-λ × ΔDays)
# λ = 0.00095 (默认值)
# 近期数据权重更高，适应市场变化
```

### 3.4 后处理校验

模型输出需要通过以下校验：

1. **RR 阈值**：`|rr_score| >= min_rr` (默认 1.5)
2. **止损范围**：`min_sl_pct <= sl_pct <= max_sl_pct`
3. **价格逻辑**：
   - LONG: `sl_price < entry_price < tp_price`
   - SHORT: `tp_price < entry_price < sl_price`

## 4. 配置管理

### 4.1 配置文件结构

```yaml
# config.yaml
data:
  symbol: "BTCUSDT"
  interval: "1h"
  binance:
    base_url: "https://fapi.binance.com"
    max_klines_per_request: 1500

features:
  technical:
    ema_periods: [9, 21, 50]
    rsi_period: 14

predict:
  model:
    n_estimators: 500
    max_depth: 6
  validation:
    min_rr: 1.5

strategy:
  trading:
    mode: "backtest"
    risk_amount: 100
    leverage: 20
```

### 4.2 配置优先级

1. 环境变量（最高优先级）
2. config.yaml 配置文件
3. 代码默认值（最低优先级）

## 8. 评估指标

### 8.1 系统指标

- **胜率 (Win Rate)**：盈利交易占比
- **盈亏比 (Profit Factor)**：总盈利 / 总亏损
- **最大回撤 (Max Drawdown)**：峰值到谷底的最大跌幅
- **夏普比率 (Sharpe Ratio)**：风险调整后收益
- **年化收益率 (CAGR)**：复合年增长率

### 8.2 模型指标

- **方向准确率**：预测方向与实际方向一致的比例
- **RR 实现率**：实际 RR / 预测 RR
- **止损有效性**：止损价格的合理性
- **止盈覆盖率**：达到止盈价格的比例

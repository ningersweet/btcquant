比特币日内合约量化交易系统 - 概要设计文档 (HLD)
版本号: v2.0 (含模型评估体系)
阶段: 概要设计 (High-Level Design)
日期: 2026-03-02
核心策略: 时间对齐网格交易 + 模型自适应点位预测
1. 系统概述
1.1 设计目标
构建一个逻辑极简、风控刚性、回测与实盘完全一致的自动化交易系统。系统摒弃连续的随机交易，采用“固定时间网格”触发机制，由 AI 模型直接输出方向、盈亏比、具体止损价和止盈价，并在固定时间窗口结束后强制平仓。
1.2 核心设计理念
时间刚性：开仓、平仓严格锚定交易所标准 K 线周期。预测窗口结束即交易结束，绝不恋战。
点位智能：模型不再只输出方向，而是直接学习并输出具体的关键支撑位（止损）和关键阻力位（止盈）。
架构解耦：数据、算法、策略、执行四层分离。执行层通过统一接口抽象，实现回测、模拟、实盘的代码零修改切换。
评估分层：严格区分“模型预测能力评估”与“系统交易表现评估”，确保迭代方向清晰。
2. 系统架构

3. 模块详细设计
3.1 数据服务层 (Data Service)
职责：提供标准化、时间对齐的数据供给。
时间对齐机制：将原始数据重采样至固定的时间网格（如1 小时，最好可配置），确保每根 K 线起始时间严格对应 （例如 1 小时的话，那就是00 分，如果是 30 分钟的话，那就是 00 或者 30 分）。
标签生成器：在离线阶段，对于每个网格点向前查看1 个窗口，识别极值和技术结构位，生成 止损价 、 止盈价 和 盈亏比 作为训练标签。
数据集划分：严格将数据划分为训练集、验证集和测试集 (Out-of-Sample)，确保评估的客观性。
3.2 算法服务层 (Algo Service)
职责：无状态的市场预测模型。
模型类型：多输出回归模型 (Multi-output Regression)。
输入：过去 N 个周期的市场特征向量。
输出 (核心三元组)：
RR_Score：正数看涨，负数看跌，绝对值为预期盈亏比。
StopLoss_Price：模型预测的关键防守价位。
TakeProfit_Price：模型预测的关键进攻价位。
约束：模型不感知账户状态，仅基于市场数据输出。
3.3 策略服务层 (Strategy Service)
职责：有状态的业务逻辑控制与风控执行。
时钟同步：仅在网格时间点唤醒。
信号校验：过滤低置信度信号 (|RR| < 阈值)，校验 SL/TP 价格逻辑合法性。
仓位管理：根据 入场价 与 模型止损价 的距离，模型输出的盈亏比，结合单笔固定风险金额，固定杠杆为 20 倍，动态计算开仓数量和使用的保证金。
持仓监控：
价格触发：触及 SL 或 TP 立即平仓。
时间触发：到达预测窗口终点 T_end，强制市价平仓。
3.4 统一执行层 (Unified Execution Engine)
职责：屏蔽底层交易渠道差异。
统一接口：open_position, close_position, get_account_info。
多模式实现：
回测模式：内存模拟撮合，严格模拟时间窗口和 OCO 订单逻辑。
模拟盘模式：对接测试网 API。
实盘模式：对接主网 API，含安全校验。
3.5 评估与分析层 (Evaluation Layer) [新增]
职责：独立于交易流程，专门用于量化模型性能和系统表现。
数据隔离：仅在测试集或模拟盘/实盘日志上运行，不参与训练。
双维度评估：
模型评估模块：专注评估预测的准确性（方向、点位）。
系统评估模块：专注评估最终的交易绩效（资金曲线、回撤）。
4. 评估体系设计 (Evaluation Framework)
本系统将评估分为两个独立维度，防止混淆“预测准不准”和“赚不赚钱”。
4.1 维度一：模型预测能力评估 (Model Performance)
目标：评估算法服务输出的 {RR, SL, TP} 是否准确反映了市场未来的走势结构。
数据来源：测试集 (Test Set) 或 实盘原始日志。
表格
指标名称	定义与计算逻辑	评估意义	理想目标
方向命中率 (Direction Hit Rate)	信号发出后，价格先向预测方向移动超过一定幅度的比例。	衡量模型判断涨跌方向的能力。	> 55%
盈亏比实现率 (RR Realization)	实际交易盈亏比 / 模型预测 RR_Score 的平均值。	衡量模型是否过度自信或保守。若 < 1.0 说明预测过于乐观。	0.8 ~ 1.2
止损有效性 (SL Efficiency)	在亏损交易中，价格触及 SL 后立即反转（未继续大幅亏损）的比例。	衡量 SL 是否设在关键的“结构破位点”，而非随机位置。	> 60%
止盈覆盖率 (TP Coverage)	在盈利交易中，价格真正触及 TP 的比例。	衡量 TP 设置是否合理。过高说明太保守，过低说明太激进。	40% ~ 60%
价格偏差度 (Price Deviation)	(预测价格 - 实际极值价格) / 实际极值价格 的平均绝对误差。	直接衡量 SL/TP 价格预测的数值精度。	越小越好
4.2 维度二：系统交易表现评估 (System Performance)
目标：评估整个量化系统（含策略逻辑、风控、手续费、滑点）的最终盈利能力。
数据来源：回测引擎报告 或 实盘账户账单。
表格
指标名称	定义与计算逻辑	评估意义	理想目标
年化收益率 (CAGR)	考虑复利后的年化增长比率。	衡量资金增长速度。	> 20%
最大回撤 (Max Drawdown)	资金曲线从最高点下跌的最大幅度。	衡量极端风险承受能力。	< 15%
夏普比率 (Sharpe Ratio)	(年化收益 - 无风险利率) / 收益波动率。	衡量单位风险下的超额收益。	> 1.5
盈利因子 (Profit Factor)	总盈利金额 / 总亏损金额。	衡量系统的整体收支平衡能力。	> 1.5
时间止损触发率	因“时间到期”而平仓的交易占总交易数的比例。	衡量模型选时效率。过高说明模型经常“看对方向但没赶上时间”。	< 40%
胜率 (Win Rate)	盈利交易次数 / 总交易次数。	配合盈亏比使用，单独看意义有限。	> 45%
4.3 模型迭代与上线标准
只有同时满足以下条件，新模型方可上线：
模型维度：测试集上的 盈亏比实现率 在 0.8~1.2 之间，且 方向命中率 优于旧模型。
系统维度：测试集回测的 最大回撤 小于 15%，且 盈利因子 > 1.5。
一致性检验：模拟盘运行 2 周的系统表现，与测试集回测结果的偏差不超过 ±20%。
5. 核心业务流程
5.1 交易触发流程
等待网格：时间到达下一个网格点（如 10:30:00）。
数据准备：获取已闭合的历史 K 线数据。
模型推理：获得 {RR, SL, TP}。
决策校验：检查 RR 阈值及 SL/TP 逻辑合法性。
执行开仓：计算仓位，发送开仓指令，记录 截止时间。
5.2 持仓管理与平仓流程
实时监听：监听市场价格和时间。
价格触发：触及 SL 或 TP -> 立即平仓。
时间触发：时间 >= 截止时间 -> 强制市价平仓。
日志记录：记录实际成交价、平仓原因、实际盈亏比，供评估层使用。
6. 风险控制设计
表格
风险维度	控制措施	实施位置
时间风险	刚性时间止损：窗口结束强制平仓	策略服务
资金风险	单笔固定风险：根据止损距离动态调整仓位	策略服务
信号风险	阈值过滤：盈亏比低于设定值不交易	策略服务
逻辑风险	价格合法性校验：检查 SL/TP 与当前价的大小关系	策略服务
极端风险	最大止损限制：系统强制限制最大亏损幅度 (如 3%)	执行层/策略层
过拟合风险	独立测试集验证：严禁使用测试集调参	评估层
7. 技术栈规划
开发语言: Python 3.9+
数据处理: Pandas, NumPy
机器学习: Scikit-learn, XGBoost / LightGBM (支持多输出)
数据存储: Parquet (历史数据), SQLite (交易记录)
交易所交互: CCXT 或 Binance Connector
可视化: Matplotlib / Plotly (用于绘制资金曲线和评估报表)
8. 实施路线图
Phase 1: 数据与标签工程 (Week 1)
完成历史数据清洗、时间对齐。
编写标签生成脚本，构造 SL, TP, RR 标签。
关键: 严格划分训练集、验证集、测试集。
Phase 2: 模型训练与评估开发 (Week 2)
训练多输出回归模型。
关键: 开发评估层代码，实现上述“模型维度”和“系统维度”的所有指标计算。
在测试集上运行评估，确立基准线 (Baseline)。
Phase 3: 回测与模拟验证 (Week 3)
开发支持时间止损和 OCO 的回测引擎。
运行全周期回测，生成“系统维度”绩效报告。
部署模拟盘，进行前瞻性验证，对比回测与模拟盘数据差异。
Phase 4: 实盘部署 (Week 4)
小资金实盘启动。
每日自动生成评估报表，监控 盈亏比实现率 等关键指标。
若指标偏离基准线，立即暂停交易并排查。
